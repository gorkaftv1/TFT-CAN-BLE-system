\section{Arquitectura general del sistema}
\label{sec:arquitectura}

El sistema desarrollado se articula en torno a tres nodos de procesamiento independientes que se comunican mediante dos canales físicos distintos: el bus CAN entre el simulador Arduino y la herramienta de diagnóstico, y Bluetooth Low Energy entre la herramienta de diagnóstico y la aplicación móvil. La figura~\ref{fig:arch_general} ilustra la arquitectura global del sistema.

\begin{figure}[!htbp]
\centering
% \includegraphics[width=\textwidth]{Ilustraciones/fig_arch_general.png}
\caption{Arquitectura general del sistema de diagnóstico del vehículo.}
\label{fig:arch_general}
\end{figure}

El \textbf{simulador de ECU} (Arduino MKR con shield CAN) reproduce el comportamiento de una unidad de control electrónico vehicular sobre el bus CAN a 500\,kbps. Responde a peticiones OBD-II en los cuatro modos implementados, genera datos de telemetría mediante un modelo físico simplificado del motor, e introduce ruido controlado en el bus para validar la robustez del sistema.

La \textbf{herramienta de diagnóstico} (Raspberry Pi con controlador MCP2515) actúa como nodo central del sistema: implementa la pila de protocolos completa (ISO-TP + OBD-II), persiste los datos en SQLite y expone el servicio BLE GATT hacia la aplicación móvil. Internamente opera con dos hilos de ejecución concurrentes: el hilo de monitorización continua y el hilo de atención a comandos BLE.

La \textbf{aplicación móvil} (React Native) se conecta a la Raspberry Pi vía BLE e interactúa con el sistema a través de comandos JSON sobre el servicio Nordic UART Service. Ofrece una interfaz de usuario estructurada en cinco pantallas funcionales.

La separación de responsabilidades entre los tres componentes permite que cada uno sea sustituido o extendido de forma independiente. En particular, la herramienta Python puede operar con un transporte simulado (\textit{mock}) sin necesidad de hardware CAN, y la aplicación móvil incorpora un adaptador simulado que permite su desarrollo sin conexión Bluetooth activa.

\section{Proceso de desarrollo iterativo}
\label{sec:desarrollo_iterativo}

El sistema se construyó de forma incremental en cuatro fases, cada una verificable de forma independiente antes de añadir la siguiente capa de funcionalidad. Este enfoque permitió detectar problemas en la capa de transporte antes de añadir la lógica de protocolo, y validar la lógica de protocolo antes de integrar la interfaz de usuario.

% ----------------------------------------------------------------------
\subsection{Fase 1: Validación del enlace CAN}
\label{sec:fase1}

El primer paso fue verificar que los dos nodos hardware —Arduino MKR y Raspberry Pi con CAN HAT RS485— podían intercambiar tramas CAN correctamente antes de introducir ninguna lógica de protocolo.

\subsubsection*{Arduino — transmisión básica CAN}

El sketch inicial se apoyó en la librería \texttt{arduino-CAN} de Sandeep Mistry, que abstrae el controlador MCP2515 vía SPI. La inicialización del bus se reduce a una sola llamada, como se muestra en el Listado~A del Anexo~A:

\begin{itemize}
    \item \texttt{CAN.begin(CAN\_SPEED)} inicializa el controlador a 500\,kbps según la constante definida en \texttt{ecu\_protocol.h} (\texttt{\#define CAN\_SPEED 500E3}).
    \item Cada trama se envía con \texttt{CAN.beginPacket(id)}, \texttt{CAN.write(data, len)} y \texttt{CAN.endPacket()}.
    \item El Serial a 115200\,baud reporta el resultado de cada operación, lo que permite supervisar el comportamiento desde el IDE de Arduino.
\end{itemize}

\begin{figure}[!htbp]
\centering
% \includegraphics[width=0.90\textwidth]{Ilustraciones/fase1_arduino_serial.png}
\caption{Arduino IDE — monitor serie durante la fase de validación CAN. Se aprecia la confirmación de inicialización del bus (\texttt{[OK] CAN Bus at 500 kbps}) y el log de cada trama enviada.}
\label{fig:fase1_arduino_serial}
\end{figure}

\subsubsection*{Raspberry Pi — recepción y herramientas SocketCAN}

En la Raspberry Pi, la interfaz CAN se levanta mediante el subsistema SocketCAN de Linux:

\begin{verbatim}
sudo ip link set can0 up type can bitrate 500000
\end{verbatim}

Las utilidades \texttt{candump} y \texttt{cansend} del paquete \texttt{can-utils} permitieron verificar la recepción de tramas y enviar peticiones manuales sin necesidad de ningún código Python:

\begin{verbatim}
candump can0
cansend can0 7DF#0201050000000000
\end{verbatim}

La salida de \texttt{candump} confirma el identificador, la longitud y los bytes de datos de cada trama recibida, lo que permitió verificar que el filtro de aceptación del controlador MCP2515 estaba correctamente configurado.

\begin{figure}[!htbp]
\centering
% \includegraphics[width=0.90\textwidth]{Ilustraciones/fase1_pi_candump.png}
\caption{Sesión SSH en la Raspberry Pi mostrando la salida de \texttt{candump can0} y una petición manual con \texttt{cansend} durante la validación del enlace CAN.}
\label{fig:fase1_pi_candump}
\end{figure}


% ----------------------------------------------------------------------
\subsection{Fase 2: Primer prototipo — simulador ECU y CLI de diagnóstico}
\label{sec:fase2}

Con el enlace CAN validado se desarrolló el primer prototipo funcional: el Arduino comenzó a responder a peticiones OBD-II reales, y la Raspberry Pi pasó de \texttt{candump} a ejecutar una herramienta Python estructurada.

\subsubsection*{Arduino — respuesta a peticiones OBD-II}

El simulador incorporó la función \texttt{processCANMessage()} que analiza el modo de servicio (primer byte de datos) y el PID solicitado (segundo byte) para construir la respuesta en el formato ISO-TP Single Frame. Los modos implementados en esta fase fueron \texttt{0x01} (datos en vivo, PIDs básicos) y \texttt{0x03} (lectura de DTCs preconfigurados).

La constante \texttt{ECU\_CAN\_ID = 0x7E0} define el identificador de petición escuchado, y \texttt{ECU\_RESPONSE\_ID = 0x7E8} el identificador de respuesta emitido, ambos definidos en \texttt{ecu\_protocol.h} en concordancia con el estándar SAE J1979.

\begin{figure}[!htbp]
\centering
% \includegraphics[width=0.90\textwidth]{Ilustraciones/fase2_arduino_serial.png}
\caption{Monitor serie del Arduino durante la Fase 2: cada petición recibida (\texttt{[RX]}) y su respuesta (\texttt{[TX]}) se registran indicando modo, PID y bytes de datos.}
\label{fig:fase2_arduino_serial}
\end{figure}

\subsubsection*{Raspberry Pi — pila ISO-TP y CLI}

En la Raspberry Pi se introdujeron las librerías \texttt{python-can}~\cite{Twiss23-pythoncan} y \texttt{can-isotp}~\cite{Menager23-isotp}, que gestionan respectivamente el acceso al bus SocketCAN y la segmentación/reensamblado ISO-TP. La clase \texttt{IsoTpTransport} encapsula ambas en la interfaz \texttt{ITransport} del sistema:

\begin{itemize}
    \item \texttt{isotp.Address} configura los identificadores de tester (\texttt{0x7E0}) y ECU (\texttt{0x7E8}) en modo Normal 11 bits.
    \item \texttt{isotp.CanStack} gestiona de forma automática los Flow Control frames y el reensamblado de respuestas multi-frame.
    \item El timeout de recepción (\texttt{timeout=5.0\,s}) absorbe la latencia de la fase de ruido sin generar falsos errores.
\end{itemize}

Sobre este transporte se desarrolló \texttt{cli.py}, un menú interactivo que permitió ejecutar consultas OBD-II desde la línea de comandos sin necesidad de la aplicación móvil. El menú inicial ofrecía dos operaciones principales:

\begin{itemize}
    \item \textbf{Opción 1 — Live data}: consulta instantánea de los cinco PIDs más relevantes (RPM, temperatura, velocidad, posición del acelerador, carga del motor).
    \item \textbf{Opción 3 — Read DTCs}: lectura de la lista de códigos de avería almacenados en la ECU simulada.
\end{itemize}

\centering
% \includegraphics[width=0.90\textwidth]{Ilustraciones/fase2_cli_basic.png}
\caption{CLI de diagnóstico (\texttt{cli.py}) en la Fase 2: menú interactivo con opciones de live data y lectura de DTCs, ejecutado en la Raspberry Pi vía SSH.}
\label{fig:fase2_cli}
\end{figure}

\subsection{Fase 3: Simulación avanzada, monitorización y persistencia}
\label{sec:fase3}

La tercera fase añadió tres mejoras simultáneas: el simulador Arduino incorporó mecanismos de ruido controlado e interrupciones hardware, y la herramienta Python extendió el CLI con monitorización continua y logging en SQLite.

\subsubsection*{Arduino — ruido CAN e interrupciones}

Los parámetros de ruido se centralizaron en \texttt{ecu\_protocol.h} como constantes compiladas:

\begin{itemize}
    \item \texttt{CAN\_NOISE\_DROP\_PCT = 2}: probabilidad de descarte de petición (simulación de colisión o degradación de señal).
    \item \texttt{CAN\_NOISE\_NRC\_PCT = 1}: probabilidad de respuesta negativa UDS con NRC \texttt{0x22} (\textit{conditionsNotCorrect}).
    \item \texttt{CAN\_NOISE\_LATENCY\_MIN\_MS = 1}, \texttt{CAN\_NOISE\_LATENCY\_MAX\_MS = 15}: retardo aleatorio antes de procesar cada petición.
    \item \texttt{CAN\_NOISE\_BG\_TRAFFIC\_PCT = 30}: probabilidad por ciclo de emitir una trama de tráfico de fondo con identificadores \texttt{0x280}, \texttt{0x480}, \texttt{0x320} o \texttt{0x520}.
\end{itemize}

Adicionalmente, se implementaron dos interrupciones hardware para simular eventos espontáneos del vehículo:

\begin{itemize}
    \item \textbf{Pin D7 (ignición)}: la ISR \texttt{onIgnitionButton()} activa el flag \texttt{ignitionPressed}; la función \texttt{handleIgnitionButton()} en el bucle principal conmuta el estado del motor y ajusta las RPM al ralentí (850\,rpm) o a cero.
    \item \textbf{Pin D6 (fallo DTC)}: la ISR \texttt{onDtcFaultButton()} activa \texttt{dtcFaultPressed}; \texttt{handleDtcFaultButton()} selecciona aleatoriamente uno de los 12 DTCs del pool, comprueba duplicados e inserta el código en \texttt{dtcList[]}.
\end{itemize}

El patrón ISR mínima + debounce en bucle principal garantiza que no se ejecuten operaciones bloqueantes (\texttt{Serial}, \texttt{delay}, acceso al bus CAN) dentro de la rutina de interrupción.

\centering

\subsubsection*{Raspberry Pi — monitorización continua y SQLite}

La herramienta Python extendió el CLI con dos nuevas capacidades:

\textbf{Monitorización continua} (\texttt{live\_data\_monitor.py}): la clase \texttt{LiveDataMonitor} ejecuta un hilo \textit{daemon} que itera sobre el conjunto de PIDs configurado, envía cada petición ISO-TP, decodifica la respuesta e invoca un callback \texttt{on\_sample} con el valor y unidades. El periodo de muestreo por defecto es 500\,ms. El hilo se inicia y detiene a través de los métodos \texttt{start()} y \texttt{stop()}, o mediante el protocolo de contexto \texttt{with}:

\begin{verbatim}
with monitor:
    input("Press Enter to stop")
\end{verbatim}

\textbf{Logging SQLite} (\texttt{sqlite\_logger.py}): \texttt{SqliteDataLogger} persiste cada muestra en la tabla \texttt{samples} (timestamp, PID, valor, unidad y trama hexadecimal) dentro de una base de datos SQLite en modo WAL, que permite escrituras concurrentes desde el hilo de monitorización sin bloqueos.

El CLI incorporó la opción 6 (\textit{Live monitor}) que arranca el monitor y actualiza la pantalla en tiempo real, y la opción 2 (\textit{Extended live data}) que interroga todos los PIDs disponibles en una sola pasada.

\subsection{Fase 4: Integración BLE y validación en vehículo real}
\label{sec:fase4}

La fase final añadió el canal de comunicación inalámbrico entre la Raspberry Pi y el teléfono móvil, y trasladó el sistema del banco de pruebas a un vehículo real mediante un cable OBD-II.

\subsubsection*{Raspberry Pi — servidor BLE GATT}

El servidor BLE se implementó en \texttt{bluetooth\_server.py} mediante la librería \texttt{bless}, que expone una interfaz asíncrona (\texttt{asyncio}) sobre el stack BlueZ de Linux. El servidor anuncia el \textbf{Nordic UART Service (NUS)} con sus dos características estándar:

\begin{itemize}
    \item \textbf{RX} (\texttt{0x6E400002-...}): característica de escritura. Recibe los comandos JSON del cliente móvil.
    \item \textbf{TX} (\texttt{0x6E400003-...}): característica de notificación. Envía las respuestas JSON al cliente.
\end{itemize}

El manejador \texttt{\_on\_write()} acumula los chunks recibidos en un buffer hasta encontrar el delimitador \texttt{\textbackslash n}, deserializa el JSON y despacha el comando al \texttt{BtCommandHandler}, que orquesta la sesión de diagnóstico y devuelve el resultado como objeto JSON serializado. Las respuestas que superan el MTU negociado (240 bytes) se fragmentan automáticamente en múltiples notificaciones.

El servidor implementa un mecanismo de \textit{watchdog} que detecta la desconexión del cliente (inactividad superior a 15\,s en RX o 20\,s en TX) y reinicia el estado de sesión, y un \textit{heartbeat} periódico cada 8\,s para mantener el enlace activo durante sesiones de monitorización prolongadas.

\subsubsection*{Protocolo de comunicación JSON}

Los comandos intercambiados entre la aplicación y la Raspberry Pi siguen un protocolo JSON sobre NUS. Los comandos principales son:

\begin{itemize}
    \item \verb|{"cmd": "auth", "token": "..."}|: autenticación de sesión (primer comando obligatorio).
    \item \verb|{"cmd": "query", "pid": "0C"}|: consulta puntual de un PID del modo 0x01.
    \item \verb|{"cmd": "read_dtcs"}| / \verb|{"cmd": "clear_dtcs"}|: gestión de averías.
    \item \verb|{"cmd": "monitor_start", "pids": ["0C","0D","05"]}|: inicio de monitorización continua con notificaciones periódicas.
    \item \verb|{"cmd": "get_sessions"}| / \verb|{"cmd": "get_samples", "session_id": N}|: consulta del historial SQLite.
\end{itemize}

\subsubsection*{Aplicación móvil — React Native + react-native-ble-plx}

La aplicación móvil implementa la comunicación BLE mediante la librería \texttt{react-native-ble-plx}, que proporciona acceso al stack BLE nativo de Android e iOS. Las operaciones clave son:

\begin{itemize}
    \item \textbf{Escaneo y conexión}: \texttt{manager.startDeviceScan()} filtra por el nombre del dispositivo (\texttt{SEAT\_DIAG}) y \texttt{device.connect()} establece el enlace físico. Tras la conexión se negocia el MTU, se descubren servicios y se activa la suscripción a notificaciones TX.
    \item \textbf{Envío de comandos}: \texttt{writeCharacteristicWithResponse()} garantiza la entrega del JSON de comando antes de esperar la respuesta.
    \item \textbf{Recepción de notificaciones}: \texttt{monitorCharacteristicForDevice()} invoca un callback con cada fragmento recibido; el adaptador los reasambla antes de procesar el JSON.
\end{itemize}

Las cinco pantallas de la aplicación se describen en detalle en la Sección~\ref{sec:app_movil}.
