\section{Arquitectura general del sistema}
\label{sec:arquitectura}

El sistema desarrollado se articula en torno a tres nodos de procesamiento independientes que se comunican mediante dos canales físicos distintos: el bus CAN entre el simulador Arduino y la herramienta de diagnóstico, y Bluetooth Low Energy entre la herramienta de diagnóstico y la aplicación móvil. La figura~\ref{fig:arch_general} ilustra la arquitectura global del sistema.

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{Ilustraciones/arch_general.png}
\caption{Arquitectura general del sistema de diagnóstico del vehículo.}
\label{fig:arch_general}
\end{figure}

El \textbf{simulador de ECU} (Arduino MKR con shield CAN) reproduce el comportamiento de una unidad de control electrónico vehicular sobre el bus CAN a 500\,kbps. Responde a peticiones OBD-II en los cuatro modos implementados (\texttt{0x01}, \texttt{0x03}, \texttt{0x04} y \texttt{0x09}) y a dos servicios UDS (\texttt{0x10} DiagnosticSessionControl y \texttt{0x22} ReadDataByIdentifier), genera datos de telemetría mediante un modelo físico simplificado del motor, y permite inyectar averías (DTCs) mediante un pulsador hardware. Opcionalmente puede activarse ruido en los valores de los sensores y la emisión de tramas de difusión periódicas, ambos desactivados por defecto.

La \textbf{herramienta de diagnóstico} (Raspberry Pi con controlador MCP2515) actúa como nodo central del sistema: implementa la pila de protocolos completa (ISO-TP + OBD-II + UDS), persiste los datos en SQLite y expone el servicio BLE GATT hacia la aplicación móvil. Internamente opera con dos hilos de ejecución concurrentes: el hilo de monitorización continua y el hilo de atención a comandos BLE.

La \textbf{aplicación móvil} (React Native) se conecta a la Raspberry Pi vía BLE e interactúa con el sistema a través de comandos JSON sobre el servicio Nordic UART Service. Ofrece una interfaz de usuario estructurada en cinco pantallas funcionales.

La separación de responsabilidades entre los tres componentes permite que cada uno sea sustituido o extendido de forma independiente. En particular, la herramienta Python puede operar con un transporte simulado (\textit{mock}) sin necesidad de hardware CAN, y la aplicación móvil incorpora un adaptador simulado que permite su desarrollo sin conexión Bluetooth activa.

Mientras que la figura~\ref{fig:arch_general} ofrece una vista estática de los nodos y sus enlaces, la figura~\ref{fig:obd2_cycle} muestra el flujo dinámico de una petición concreta a través del sistema completo: desde el comando JSON emitido por la aplicación móvil, su traducción a una trama ISO-TP/CAN en la Raspberry Pi, la respuesta de la ECU simulada y el camino de vuelta de los datos decodificados hasta el cuadro de mando de la aplicación.

\begin{figure}[H]
\centering
\includegraphics[width=0.9\textwidth]{Ilustraciones/obd2_cycle.png}
\caption{Flujo extremo a extremo de una petición OBD-II en el sistema desarrollado: aplicación móvil → BLE → Raspberry Pi → ISO-TP/CAN → ECU, y el camino de vuelta con los datos decodificados.}
\label{fig:obd2_cycle}
\end{figure}

\section{Dependencias y entorno de desarrollo}
\label{sec:dependencias}

Cada uno de los tres componentes se apoya en un conjunto reducido de dependencias de terceros, seleccionadas por implementar estándares abiertos. A continuación se describe el entorno y las librerías de cada nodo, así como el uso concreto que se hace de cada una.

\medskip
\noindent\textbf{Simulador de ECU (Arduino)}

El firmware se desarrolla en C++ sobre el \textit{core} de Arduino y se compila con el Arduino IDE. Su única dependencia externa es:

\begin{itemize}
    \item \textbf{\texttt{arduino-CAN}} de Sandeep Mistry~\cite{ArduinoCAN_lib}: abstrae el controlador CAN MCP2515~\cite{Microchip_MCP2515} a través del bus SPI. Se utiliza para inicializar el bus a 500\,kbps (\texttt{CAN.begin}) y para enviar y recibir las tramas crudas de 8 bytes (\texttt{beginPacket}/\texttt{write}/\texttt{endPacket}, \texttt{parsePacket}/\texttt{read}). Toda la lógica ISO-TP, OBD-II y UDS se implementa manualmente sobre estas primitivas.
\end{itemize}

\medskip
\noindent\textbf{Herramienta de diagnóstico (Raspberry Pi, Python 3)}

La herramienta se ejecuta sobre Python 3 en Raspberry Pi OS, apoyándose en el subsistema SocketCAN del kernel Linux~\cite{LinuxKernel_ISOTP} para el acceso al bus a través del CAN HAT~\cite{Waveshare_RS485CANHAT}. Sus dependencias (recogidas en \texttt{requirements.txt}) son:

\begin{itemize}
    \item \textbf{\texttt{python-can}}~\cite{Twiss23-pythoncan}: proporciona la abstracción de alto nivel sobre la interfaz SocketCAN (\texttt{can.Bus}). Se usa como capa de acceso al bus por debajo de ISO-TP.
    \item \textbf{\texttt{can-isotp}}~\cite{Menager23-isotp}: implementa el protocolo de transporte ISO 15765-2. Se usa (\texttt{isotp.CanStack}) para segmentar y reensamblar de forma transparente los mensajes que superan los 8 bytes de una trama CAN, gestionando \textit{First/Consecutive Frames} y \textit{Flow Control}.
    \item \textbf{\texttt{bless}}~\cite{Bless_lib}: servidor GATT BLE multiplataforma sobre \texttt{asyncio} y BlueZ. Se usa para publicar el servicio Nordic UART Service y atender las escrituras y notificaciones del cliente móvil.
\end{itemize}

La persistencia (\texttt{sqlite3}), la concurrencia (\texttt{threading}, \texttt{asyncio}) y el registro (\texttt{logging}) se resuelven con la biblioteca estándar de Python, sin dependencias adicionales.

\medskip
\noindent\textbf{Aplicación móvil (React Native)}

La aplicación se construye con React Native sobre el \textit{framework} Expo~\cite{Expo_docs}, lo que permite una única base de código para Android e iOS. Sus dependencias principales son:

\begin{itemize}
    \item \textbf{\texttt{react-native-ble-plx}}~\cite{ReactNativeBLEPLX}: cliente BLE nativo. Se usa para el escaneo, la conexión, la negociación de MTU, la escritura en la característica RX y la suscripción a notificaciones de la característica TX del servicio NUS.
    \item \textbf{\texttt{zustand}}~\cite{Zustand_lib}: gestión de estado global ligera. Se usa para los almacenes que mantienen el estado de conexión, la telemetría, los DTCs, las sesiones y la configuración.
    \item \textbf{@\texttt{react-navigation}} (\texttt{bottom-tabs}): navegación por pestañas de la interfaz.
    \item \textbf{@\texttt{react-native-async-storage}}: persistencia local de preferencias; \textbf{\texttt{expo-file-system}} y \textbf{\texttt{expo-sharing}}: exportación del registro de la aplicación.
\end{itemize}

\section{Proceso de desarrollo iterativo}
\label{sec:desarrollo_iterativo}

El sistema se construyó de forma incremental en cuatro fases, cada una verificable de forma independiente antes de añadir la siguiente capa de funcionalidad. Este enfoque permitió detectar problemas en la capa de transporte antes de añadir la lógica de protocolo, y validar la lógica de protocolo antes de integrar la interfaz de usuario.

% ----------------------------------------------------------------------
\subsection{Fase 1: Validación del enlace CAN}
\label{sec:fase1}

El primer paso fue verificar que los dos nodos hardware —Arduino MKR y Raspberry Pi con CAN HAT RS485— podían intercambiar tramas CAN correctamente antes de introducir ninguna lógica de protocolo.

\medskip
\noindent\textbf{Arduino — transmisión básica CAN}

\begin{figure}[H]
\centering
\includegraphics[width=\textwidth]{Ilustraciones/iter_1_ard.png}
\caption{Arduino IDE — monitor serie durante la fase de validación CAN. Se aprecia la confirmación de inicialización del bus (\texttt{[OK] CAN Bus at 500 kbps}) y el log de cada trama enviada.}
\label{fig:fase1_arduino_serial}
\end{figure}

El sketch inicial se apoyó en la librería \texttt{arduino-CAN} de Sandeep Mistry, que abstrae el controlador MCP2515 vía SPI. La inicialización del bus se reduce a una sola llamada, como se resume a continuación:

\begin{itemize}
    \item \texttt{CAN.begin(CAN\_SPEED)} inicializa el controlador a 500\,kbps según la constante definida en \texttt{ecu\_protocol.h} (\texttt{\#define CAN\_SPEED 500E3}).
    \item Cada trama se envía con \texttt{CAN.beginPacket(id)}, \texttt{CAN.write(data, len)} y \texttt{CAN.endPacket()}.
    \item El Serial a 115200\,baud reporta el resultado de cada operación, lo que permite supervisar el comportamiento desde el IDE de Arduino.
\end{itemize}

\medskip
\noindent\textbf{Raspberry Pi — recepción y herramientas SocketCAN}

\begin{figure}[h]
\centering
\includegraphics[width=\textwidth]{Ilustraciones/iter_1_pi.png}
\caption{Sesión SSH en la Raspberry Pi mostrando la salida de \texttt{candump can0} y una petición manual con \texttt{cansend} durante la validación del enlace CAN.}
\label{fig:fase1_pi_candump}
\end{figure}

En la Raspberry Pi, la interfaz CAN se levanta mediante el subsistema SocketCAN de Linux:

\begin{verbatim}
sudo ip link set can0 up type can bitrate 500000
\end{verbatim}

Las utilidades \texttt{candump} y \texttt{cansend} del paquete \texttt{can-utils} permitieron verificar la recepción de tramas y enviar peticiones manuales sin necesidad de ningún código Python:

\begin{verbatim}
candump can0
cansend can0 7E0#0201050000000000
\end{verbatim}

La salida de \texttt{candump} confirma el identificador, la longitud y los bytes de datos de cada trama recibida, lo que permitió verificar que la recepción sobre el controlador MCP2515 estaba correctamente configurada.

\subsection{Fase 2: Primer prototipo — simulador ECU y CLI de diagnóstico}
\label{sec:fase2}

Ya validado el enlace CAN se desarolla el primer prototipo funcional: el Arduino responde a peticiones OBD-II reales, y la Raspberry Pi script de python interactivo.

\medskip
\noindent\textbf{Arduino — respuesta a peticiones OBD-II}

\begin{figure}[H]
\centering
\includegraphics[width=0.35\textwidth]{Ilustraciones/iter_2_ard.png}
\caption{Monitor serie del Arduino durante la Fase 2: cada petición recibida (\texttt{[RX]}) y su respuesta (\texttt{[TX SF]}) se registran indicando modo, PID y bytes de datos.}
\label{fig:fase2_arduino_serial}
\end{figure}

Se incropora la función \texttt{processCANMessages()} que analiza el modo de servicio (segundo byte de datos, tras el byte PCI de ISO-TP) y el PID solicitado para construir la respuesta en el formato ISO-TP Single Frame. Los modos implementados en esta fase fueron \texttt{0x01} (datos en vivo, PIDs básicos) y \texttt{0x03} (lectura de DTCs preconfigurados).

Además se añadió el modo \texttt{0x09} (información del vehículo), cuyo PID \texttt{0x02} devuelve el VIN de 17 caracteres. Al superar la respuesta los 7 bytes útiles de un Single Frame, la función \texttt{sendVINMultiFrame()} implementa la transmisión ISO-TP multi-frame del lado emisor: envía primero un First Frame (\texttt{FF}, PCI \texttt{0x1X}) con la longitud total de la carga, espera el Flow Control del escáner mediante \texttt{waitForFlowControl()} y a continuación emite los Consecutive Frames (\texttt{CF}, PCI \texttt{0x2X}) con número de secuencia incremental y una separación temporal entre tramas. Es la primera segmentación manual del protocolo en el Arduino, ya que las librerías ISO-TP residen solo en el lado de la Raspberry Pi.

La constante \texttt{ECU\_CAN\_ID = 0x7E0} define el identificador de petición escuchado, y \texttt{ECU\_RESPONSE\_ID = 0x7E8} el identificador de respuesta emitido, ambos definidos en \texttt{ecu\_protocol.h} en concordancia con el direccionamiento físico OBD-II sobre CAN.


\noindent\textbf{Raspberry Pi — pila ISO-TP y CLI}

\begin{figure}[H]
\centering
\includegraphics[height=6.5cm]{Ilustraciones/iter_2_pi.png}
\caption{CLI de diagnóstico (\texttt{cli.py}) en la Fase 2: menú interactivo con opciones de live data y lectura de DTCs, ejecutado en la Raspberry Pi vía SSH.}
\label{fig:fase2_cli}
\end{figure}

En la Raspberry Pi se introdujeron las librerías \texttt{python-can}~\cite{Twiss23-pythoncan} y \texttt{can-isotp}~\cite{Menager23-isotp}, que gestionan respectivamente el acceso al bus SocketCAN y la segmentación/reensamblado ISO-TP. La clase \texttt{IsoTpTransport} encapsula ambas en la interfaz \texttt{ITransport} del sistema:

\begin{lstlisting}[language=Python, caption={Configuración de la pila ISO-TP en \texttt{IsoTpTransport}}]
# Identificadores escáner (0x7E0) y ECU (0x7E8) en modo Normal 11 bits
self._address = isotp.Address(
    addressing_mode=isotp.AddressingMode.Normal_11bits,
    txid=tx_id,   # 0x7E0 — peticiones del escáner
    rxid=rx_id,   # 0x7E8 — respuestas de la ECU
)

# CanStack gestiona de forma automática los Flow Control frames
# y el reensamblado de las respuestas multi-frame
self._stack = isotp.CanStack(self._bus, address=self._address,
                             params=self._params)

# El timeout de recepción absorbe la latencia de la transmisión
# multi-frame sin generar falsos errores de comunicación
deadline = time.monotonic() + self._timeout   # timeout = 5.0 s
while time.monotonic() < deadline:
    self._stack.process()
    if self._stack.available():
        return self._stack.recv()
\end{lstlisting}

Sobre este transporte se desarrolló \texttt{cli.py}, un menú interactivo que permitió ejecutar consultas OBD-II desde la línea de comandos sin necesidad de la aplicación móvil. El menú inicial ofrecía operaciones como:

\begin{itemize}
    \item \textbf{Opción 1 — Live data}: consulta instantánea de los cinco PIDs más relevantes (temperatura del refrigerante, carga, RPM, velocidad y posición del acelerador).
    \item \textbf{Opción 3 — Read DTCs}: lectura de la lista de códigos de avería almacenados en la ECU simulada.
\end{itemize}

\subsection{Fase 3: Simulación avanzada, monitorización y persistencia}
\label{sec:fase3}

La tercera fase añadió varias mejoras simultáneas: el simulador Arduino incorporó interrupciones hardware y los servicios UDS (ISO 14229-1), y la herramienta Python extendió el CLI con monitorización continua, logging en SQLite y un cliente UDS.

\medskip
\noindent\textbf{Arduino — interrupciones hardware, servicios UDS y ruido opcional}

\begin{figure}[H]
\centering
\includegraphics[height=6.5cm]{Ilustraciones/iter_3_ard.png}
\caption{Monitor serie del Arduino en la Fase 3: se aprecian los eventos de ignición (\texttt{[IGN] Key ON — engine starting}) y la inyección de DTCs por interrupción hardware (\texttt{[DTC] Fault injected}).}
\label{fig:fase3_arduino_noise}
\end{figure}

Se implementaron dos interrupciones hardware para simular eventos espontáneos del vehículo. Ambas siguen el patrón de ISR mínima con histéresis temporal por \texttt{millis()} (300\,ms) para el antirrebote, y la lógica de efecto se ejecuta fuera de la ISR, en el bucle principal, donde es seguro realizar operaciones bloqueantes (\texttt{Serial}, acceso al bus CAN):

\begin{itemize}
    \item \textbf{Pin D5 (ignición, \texttt{ENGINE\_START\_PIN})}: la ISR \texttt{onIgnitionPress()} incrementa el contador \texttt{ignitionFlag}; \texttt{handleIgnitionButton()} conmuta el estado del motor y fija las RPM al ralentí (850\,rpm) o a cero.
    \item \textbf{Pin D4 (fallo DTC, \texttt{DTC\_FAULT\_PIN})}: la ISR \texttt{onDTCPress()} incrementa \texttt{dtcFlag}; \texttt{handleDTCButton()} inyecta secuencialmente uno de los tres códigos del conjunto \texttt{DTC\_FAULT\_SET} (\texttt{P0300}, \texttt{P0171}, \texttt{P0420}) en \texttt{dtcList[]} y activa el testigo \texttt{checkEngine}. Una cuarta pulsación, cuando ya hay tres averías activas, borra todas.
\end{itemize}

Adicionalmente, el simulador admite dos mecanismos opcionales de variabilidad, controlados por sendas macros en \texttt{ecu\_protocol.h} y \textbf{desactivados por defecto} para obtener un bus determinista durante la depuración:

\begin{itemize}
    \item \texttt{ADD\_NOISE} (por defecto \texttt{0}): cuando se activa, añade \textit{jitter} pseudoaleatorio a los valores de los sensores en cada ciclo de actualización (RPM, temperaturas, MAF, tensión de batería y \textit{fuel trim}), con amplitudes por sensor parametrizadas (\texttt{NOISE\_RPM\_MAX}, etc.).
    \item \texttt{BROADCAST\_ENABLE} (por defecto \texttt{0}): cuando se activa, el simulador emite cada \texttt{BROADCAST\_INTERVAL\_MS} (100\,ms) tres tramas CAN no solicitadas con identificadores \texttt{0x280} (RPM/carga), \texttt{0x320} (velocidad) y \texttt{0x3D0} (refrigerante/acelerador/batería), imitando el tráfico periódico de una ECU real.
\end{itemize}

El simulador incorporó además dos servicios UDS (ISO 14229-1) sobre la misma pila ISO-TP: \textit{DiagnosticSessionControl} (\texttt{0x10}), que conmuta entre la sesión por defecto y la extendida, y \textit{ReadDataByIdentifier} (\texttt{0x22}), que expone DID estándar (VIN, número de serie y versión de software) y DID propietarios de telemetría accesibles solo en sesión extendida. Su implementación se detalla en el apartado~\ref{sec:obd_uds_sim}.

\medskip
\noindent\textbf{Raspberry Pi — monitorización continua, SQLite y cliente UDS}

La herramienta Python extendió el CLI con tres nuevas capacidades:

\textbf{Monitorización continua} (\texttt{live\_data\_monitor.py}): la clase \texttt{LiveDataMonitor} ejecuta un hilo \textit{daemon} que itera sobre el conjunto de PIDs configurado, envía cada petición ISO-TP, decodifica la respuesta e invoca un callback \texttt{on\_sample} con un objeto \texttt{MonitorSample} (PID, nombre, valor, unidad y \textit{timestamp} monotónico) por cada PID decodificado con éxito. Los errores se notifican por separado mediante el callback opcional \texttt{on\_error(pid, exc)}. El periodo de muestreo por defecto es 500\,ms. El hilo se inicia y detiene a través de los métodos \texttt{start()} y \texttt{stop()}, o mediante el protocolo de contexto \texttt{with}:

\begin{verbatim}
with monitor:
    input("Press Enter to stop")
\end{verbatim}

\textbf{Logging SQLite} (\texttt{sqlite\_logger.py}): \texttt{SqliteDataLogger} almacena cada muestra en la tabla \texttt{samples} (PID, nombre, valor, unidad, timestamp monotónico y timestamp de pared ISO-8601) dentro de una base de datos SQLite en modo WAL, que permite escrituras concurrentes desde el hilo de monitorización sin bloqueos. Las tramas CAN en hexadecimal de cada comando se registran por separado en la tabla \texttt{commands}.

\textbf{Cliente UDS} (\texttt{uds\_session.py}): la clase \texttt{UdsSession} envuelve el \texttt{ITransport} con \texttt{UdsProtocolBuilder} y \texttt{UdsDataDecoder} para exponer dos operaciones de alto nivel del estándar ISO 14229-1: \texttt{set\_session()}, que emite \textit{DiagnosticSessionControl} (servicio \texttt{0x10}) y registra la sesión activa, y \texttt{read\_did()}, que emite \textit{ReadDataByIdentifier} (servicio \texttt{0x22}) y decodifica el valor según el registro de DIDs (\texttt{uds\_dids.py}). Este registro contiene tres DID estándar (\texttt{0xF190} VIN, \texttt{0xF18C} número de serie, \texttt{0xF189} versión de software) y ocho DID propietarios de datos en vivo (\texttt{0x2001}--\texttt{0x2008}, marcados \texttt{extended\_only}). El cliente verifica la sesión antes de enviar: si se solicita un DID propietario fuera de la sesión extendida, rechaza la petición localmente con NRC \texttt{0x7E} sin acceder al bus.

El CLI añade la opción 6 (\textit{Live monitor}) que arranca el monitor y actualiza la pantalla en tiempo real, la opción 2 (\textit{Extended live data}) que interroga todos los PIDs disponibles en una sola pasada, y las opciones 7--9 para UDS: cambio de sesión (\textit{change session}), lectura de los DID estándar y lectura de los DID extendidos.

\begin{figure}[H]
\centering
\includegraphics[height=6.5cm]{Ilustraciones/iter_3_pi.png}
\caption{CLI en modo de monitorización continua (opción 6): actualización periódica de los cinco PIDs principales con timestamp, valores decodificados y unidades, ejecutado vía SSH en la Raspberry Pi.}
\label{fig:fase3_cli_monitor}
\end{figure}

\subsection{Fase 4: Integración BLE y validación en vehículo real}
\label{sec:fase4}

La fase final añadió el canal de comunicación inalámbrico entre la Raspberry Pi y el teléfono móvil, y se comenzó con las pruebas en vehiculos reales.

\medskip
\noindent\textbf{Raspberry Pi — servidor BLE GATT}

El servidor BLE se implementó en \texttt{bluetooth\_server.py} mediante la librería \texttt{bless}, que expone una interfaz asíncrona (\texttt{asyncio}) sobre el stack BlueZ de Linux. El servidor anuncia el \textbf{Nordic UART Service (NUS)} con el nombre \texttt{diag\_tool} y sus dos características estándar:

\begin{itemize}
    \item \textbf{RX} (\texttt{6E400002-...}): característica de escritura. Recibe los comandos JSON del cliente móvil.
    \item \textbf{TX} (\texttt{6E400003-...}): característica de notificación. Envía las respuestas JSON al cliente.
\end{itemize}

El manejador \texttt{\_on\_write()} acumula los \textit{chunks} recibidos en un buffer hasta encontrar el delimitador \texttt{\textbackslash n}, deserializa el JSON y despacha el comando al \texttt{BtCommandHandler}, que orquesta la sesión de diagnóstico y devuelve el resultado como objeto JSON serializado. Las respuestas que superan el MTU del servidor (240 bytes) se fragmentan automáticamente en múltiples notificaciones.

El servidor implementa un mecanismo de \textit{watchdog} que detecta la desconexión del cliente (inactividad superior a 15\,s en RX o 20\,s en TX) y reinicia el estado de sesión, y un \textit{heartbeat} periódico cada 8\,s para mantener el enlace activo durante sesiones de monitorización prolongadas.

\begin{figure}[H]
\centering
\includegraphics[height=6.5cm]{Ilustraciones/iter_4_pi.png}
\caption{Consola de la Raspberry Pi ejecutando el servidor BLE: anuncio del servicio \texttt{diag\_tool} y registro de los comandos JSON recibidos (RX) y respuestas enviadas (TX).}
\label{fig:fase4_pi_server}
\end{figure}

\medskip
\noindent\textbf{Protocolo de comunicación JSON}

Los comandos intercambiados entre la aplicación y la Raspberry Pi siguen un protocolo JSON delimitado por saltos de línea (NDJSON) sobre NUS. Los comandos principales son:

\begin{itemize}
    \item \verb|{"cmd": "auth", "token": "..."}|: autenticación de sesión (primer comando obligatorio).
    \item \verb|{"cmd": "snapshot"}|: lectura puntual de todos los PIDs soportados del modo 0x01.
    \item \verb|{"cmd": "dtcs"}| / \verb|{"cmd": "clear_dtcs"}| / \verb|{"cmd": "vin"}|: gestión de averías y lectura del VIN.
    \item \verb|{"cmd": "monitor_start", "pids": [12,13,5]}| / \verb|{"cmd": "monitor_stop"}|: inicio y parada de monitorización continua con notificaciones periódicas.
    \item \verb|{"cmd": "sessions"}| / \verb|{"cmd": "session_samples", "session_id": N}|: consulta del historial SQLite.
    \item \verb|{"cmd": "uds_session", "session_type": 3}| / \verb|{"cmd": "uds_read_did", "did": "0xF190"}|: control de sesión UDS y lectura de DIDs.
    \item \verb|{"cmd": "probe_pids"}|: descubrimiento automático de los PIDs soportados por la ECU.
\end{itemize}

\medskip
\noindent\textbf{Aplicación móvil — React Native + react-native-ble-plx}

La aplicación móvil implementa la comunicación BLE mediante la librería \texttt{react-native-ble-plx}, que proporciona acceso al stack BLE nativo de Android e iOS. Las operaciones clave son:

\begin{itemize}
    \item \textbf{Escaneo y conexión}: \texttt{manager.startDeviceScan(null, ...)} enumera todos los dispositivos BLE cercanos (sin filtro de UUID) y el usuario selecciona el adaptador, que la Raspberry Pi anuncia con el nombre \texttt{diag\_tool}. \texttt{connectToDevice()} establece el enlace físico, a continuación se negocia el MTU, se descubren servicios y se activa la suscripción a notificaciones TX.
    \item \textbf{Envío de comandos}: \texttt{writeCharacteristicWithResponse()} garantiza la entrega del JSON de comando antes de esperar la respuesta.
    \item \textbf{Recepción de notificaciones}: \texttt{monitorCharacteristicForService()} invoca un callback con cada fragmento recibido, el adaptador los reasambla por líneas (NDJSON) antes de procesar el JSON.
\end{itemize}

\section{Simulador de ECU — Arduino MKR}
\label{sec:arduino}

El simulador de ECU es el componente que permite validar la herramienta de diagnóstico de forma independiente de un vehículo real. Se implementa sobre un Arduino MKR con shield CAN basado en el controlador MCP2515, conectado al bus CAN a 500\,kbps. El código fuente completo del simulador está disponible en el repositorio del proyecto.


\subsection{Hardware y configuración del bus CAN}

El Arduino MKR opera a 3,3\,V y dispone de comunicación SPI para el controlador MCP2515. La librería \texttt{arduino-CAN} inicializa el controlador a 500\,kbps. El simulador acepta únicamente las tramas dirigidas a la dirección física de la ECU (\texttt{0x7E0}), descartando en software cualquier otro identificador recibido.

Las constantes de configuración del protocolo se concentran en el fichero \texttt{ecu\_protocol.h}, separando la parametrización del código de lógica. Entre los identificadores definidos destacan:

\begin{itemize}
    \item \texttt{ECU\_CAN\_ID} = \texttt{0x7E0}: identificador escuchado (peticiones del escáner)
    \item \texttt{ECU\_RESPONSE\_ID} = \texttt{0x7E8}: identificador de respuesta de la ECU
    \item \texttt{ISOTP\_CF\_SEP\_MS} = 25\,ms: separación temporal entre \textit{Consecutive Frames} consecutivos
    \item \texttt{ISOTP\_FC\_TIMEOUT\_MS} = 1000\,ms: tiempo máximo de espera del \textit{Flow Control} del escáner
    \item \texttt{ISOTP\_PADDING\_BYTE} = \texttt{0xAA}: byte de relleno hasta los 8 bytes de la trama CAN
\end{itemize}


\subsection{Modelo físico del motor}

Para que los datos de telemetría devueltos por el simulador presenten un comportamiento realista y dinámico, se implementa un modelo físico simplificado del motor que actualiza sus variables de estado en cada iteración del bucle principal.

Las variables del modelo físico son:

\begin{itemize}
    \item \textbf{Posición del acelerador}: actúa como entrada del modelo y evoluciona mediante una máquina de estados de rampa automática (ralentí → subida → crucero → bajada → repetir), con incrementos de \texttt{RAMP\_STEP} (5\,\%) por ciclo hasta un máximo de \texttt{RAMP\_MAX\_THROTTLE} (65\,\%) y permanencias configurables en cada extremo (\textasciitilde{}3\,s).

    \item \textbf{RPM}: sigue un retardo de primer orden (filtro exponencial con factor $\alpha$) hacia un objetivo proporcional al acelerador, calculado como \texttt{map(throttle, 0--100, 850--5500)}, y se acota finalmente al rango 750--6500\,rpm.

    \item \textbf{Velocidad}: integra un modelo longitudinal simplificado en el que una fuerza de tracción (proporcional a RPM\,$\times$\,carga) compite con una resistencia cuadrática (proporcional a la velocidad al cuadrado), acotada a 0--180\,km/h.

    \item \textbf{Carga del motor}: derivada del acelerador mediante \texttt{map(throttle, 0--100, 15--85)}\,\%. A partir de ella se calculan el caudal MAF, el avance de encendido y las presiones de combustible y admisión.

    \item \textbf{Temperaturas de refrigerante y aceite}: siguen una curva de calentamiento incremental desde la temperatura ambiente hasta su régimen de operación (\textasciitilde{}90\,°C y \textasciitilde{}95\,°C), y se enfrían progresivamente con el motor apagado.

    \item \textbf{Tensión de batería} y \textbf{nivel de combustible}: la tensión sube con el motor en marcha (alternador) y cae en reposo; el combustible se consume a un ritmo dependiente de la carga.
\end{itemize}

El modelo se actualiza cada \texttt{SIM\_UPDATE\_INTERVAL\_DEFAULT} (200\,ms) en el bucle principal, y los valores calculados se codifican directamente en los bytes de respuesta OBD-II según las fórmulas del estándar SAE J1979 para cada PID. El Listado~\ref{lst:modelo_fisico} recoge el núcleo de este modelo: el filtro exponencial de primer orden que lleva las RPM hacia su objetivo y el modelo longitudinal que integra la velocidad a partir del balance entre fuerza motriz y resistencia aerodinámica.

\begin{lstlisting}[language=C++,caption={Núcleo del modelo físico del motor: filtro exponencial de RPM e integración de velocidad (\texttt{updateVehicleSimulation()}).},label={lst:modelo_fisico}]
// Objetivo de RPM proporcional al acelerador + filtro exponencial (retardo 1er orden)
int16_t targetRpm = (int16_t)map(vehicle.throttlePos, 0, 100, 850, 5500);
float alpha = 0.3f * ((float)dtMs / 200.0f);          // factor del filtro (depende de dt)
if (alpha > 1.0f) alpha = 1.0f;
vehicle.rpm = (uint16_t)(vehicle.rpm + alpha * (targetRpm - (int16_t)vehicle.rpm));

// Modelo longitudinal: fuerza motriz (prop. RPM*carga) menos arrastre (prop. v^2)
float force = 0.00002f * (float)vehicle.rpm * (float)vehicle.engineLoad
            - 0.003f   * (float)vehicle.speed * (float)vehicle.speed;
force *= ((float)dtMs / 200.0f);                       // integracion por delta-t
vehicle.speed = (uint8_t)constrain((int16_t)vehicle.speed + (int16_t)force, 0, 180);

// Parametros correlacionados derivados del estado del motor
vehicle.engineLoad    = (uint8_t)map(vehicle.throttlePos, 0, 100, 15, 85);
vehicle.mafFlow       = (uint16_t)((vehicle.rpm * vehicle.engineLoad) / 500);
vehicle.timingAdvance = (int8_t)map(vehicle.rpm, 800, 6000, 8, 32);

// Acotado final de cada variable a su rango fisico valido
vehicle.rpm   = constrain(vehicle.rpm,   750, 6500);
vehicle.speed = constrain(vehicle.speed,   0,  180);
\end{lstlisting}


\subsection{Implementación de los modos OBD-II y servicios UDS}
\label{sec:obd_uds_sim}

La función central \texttt{processCANMessages()} analiza el byte de modo de servicio de cada trama recibida y despacha al manejador correspondiente; en los modos 0x01 y 0x09 utiliza además el byte de PID.

\medskip
\noindent\textbf{Modo 0x01 — Datos en vivo}

Se implementan 25 PIDs de datos. La respuesta se construye en un buffer de 8 bytes con el formato OBD-II estándar: byte de longitud ISO-TP (tipo SF + longitud), byte de modo de respuesta (\texttt{0x41}), byte de PID y bytes de datos codificados.

La decodificación inversa sigue las fórmulas del estándar para cada PID. Como ejemplo, el PID \texttt{0x0C} (RPM) codifica el valor en dos bytes según:

\[
    \text{valor raw} = \text{RPM} \times 4 \quad \Rightarrow \quad
    \text{byte\_A} = \lfloor \text{valor raw} / 256 \rfloor, \quad
    \text{byte\_B} = \text{valor raw} \bmod 256
\]

Los PIDs de soporte \texttt{0x00}, \texttt{0x20} y \texttt{0x40} devuelven cada uno una máscara de bits de 32 bits que indica qué PIDs del rango siguiente están disponibles (el bit menos significativo señala la existencia del siguiente bloque), permitiendo a la herramienta de diagnóstico realizar una enumeración automática de capacidades a lo largo de los tres rangos.

\medskip
\noindent\textbf{Modo 0x03 — Lectura de DTCs}

El simulador mantiene una lista de DTCs activos (\texttt{dtcList[]}) con un contador \texttt{numStoredDTCs}. La respuesta del modo 0x03 incluye la cuenta de DTCs seguida de los pares de bytes correspondientes a cada código, codificados según el formato J2012: los dos bits más significativos del primer byte indican la categoría (00=Powertrain, 01=Chassis, 10=Body, 11=User (Network)) y los 14 bits restantes el código numérico.

\medskip
\noindent\textbf{Modo 0x04 — Borrado de DTCs}

Al recibir una petición de modo 0x04, el simulador reinicia \texttt{numStoredDTCs} a cero, desactiva el flag \texttt{checkEngine} y responde con el byte de confirmación \texttt{0x44}.

\medskip
\noindent\textbf{Modo 0x09 — Información del vehículo (VIN)}

El PID \texttt{0x02} del modo 0x09 devuelve el VIN del vehículo, una cadena ASCII de 17 caracteres. Dado que la respuesta completa ocupa 20 bytes (3 de cabecera más 17 del VIN), supera la capacidad de una Single Frame y requiere transmisión multi-frame.

La secuencia de transmisión es:
\begin{enumerate}
    \item \textbf{First Frame}: PCI \texttt{0x10 0x14} (longitud = 20 bytes), modo \texttt{0x49}, PID \texttt{0x02}, contador \texttt{0x01} y primeros 3 bytes del VIN.
    \item Espera del \textbf{Flow Control} del escáner.
    \item \textbf{Consecutive Frame 1}: SN \texttt{0x21}, bytes 4--10 del VIN.
    \item \textbf{Consecutive Frame 2}: SN \texttt{0x22}, bytes 11--17 del VIN (con padding).
\end{enumerate}

El Listado~\ref{lst:vin_multiframe} muestra la implementación real de esta secuencia en el simulador.

\begin{lstlisting}[language=C++,caption={Transmisión ISO-TP multi-frame del VIN en el simulador (\texttt{sendVINMultiFrame()}).},label={lst:vin_multiframe}]
void sendVINMultiFrame() {
  const uint8_t PAYLOAD_LEN = 3 + VIN_LENGTH;            // 3 cabecera + 17 VIN = 20
  uint8_t payload[20];
  payload[0] = MODE_09_VEHICLE_INFO + RESPONSE_SUCCESS;  // 0x49
  payload[1] = 0x02;                                     // PID
  payload[2] = 0x01;                                     // num. de elementos
  for (uint8_t i = 0; i < VIN_LENGTH; i++)
    payload[3 + i] = (uint8_t)vehicle.vin[i];

  // First Frame: PCI (0x1X XX = longitud total) + primeros 6 bytes del payload
  uint8_t ff[8];
  ff[0] = ISOTP_PCI_FF | ((PAYLOAD_LEN >> 8) & 0x0F);
  ff[1] = PAYLOAD_LEN & 0xFF;
  for (uint8_t i = 0; i < ISOTP_FF_DATA_BYTES; i++) ff[2 + i] = payload[i];
  CAN.beginPacket(ECU_RESPONSE_ID);
  for (uint8_t i = 0; i < 8; i++) CAN.write(ff[i]);
  CAN.endPacket();

  // Espera del Flow Control del escáner antes de enviar las Consecutive Frames
  if (!waitForFlowControl())
    Serial.println(F("[WARN] FC timeout - sending CFs anyway"));

  uint8_t bytesSent = ISOTP_FF_DATA_BYTES;
  uint8_t sn = 1;                                        // número de secuencia
  while (bytesSent < PAYLOAD_LEN) {
    uint8_t cf[8];
    cf[0] = ISOTP_PCI_CF | (sn & 0x0F);
    for (uint8_t i = 0; i < ISOTP_CF_DATA_BYTES; i++) {
      uint8_t idx = bytesSent + i;
      cf[1 + i] = (idx < PAYLOAD_LEN) ? payload[idx] : ISOTP_PADDING_BYTE;
    }
    CAN.beginPacket(ECU_RESPONSE_ID);
    for (uint8_t i = 0; i < 8; i++) CAN.write(cf[i]);
    CAN.endPacket();
    bytesSent += ISOTP_CF_DATA_BYTES;
    sn = (sn + 1) & 0x0F;
    if (bytesSent < PAYLOAD_LEN) delay(ISOTP_CF_SEP_MS); // separación entre tramas
  }
}
\end{lstlisting}

Esta implementación manual de la secuencia multi-frame en el simulador fue clave, dado que cualquier error en el contador de secuencia o en el timing entre tramas provocaba el rechazo de la sesión por parte de la librería \texttt{can-isotp} en la Raspberry Pi.

\medskip
\noindent\textbf{Servicios UDS (ISO 14229-1)}

Además de los cuatro modos OBD-II, el simulador implementa dos servicios UDS que permiten un diagnóstico más rico que el conjunto de PIDs estándar:

\begin{itemize}
    \item \textbf{Servicio \texttt{0x10} — DiagnosticSessionControl}: \texttt{handleMode10()} admite las subfunciones \texttt{0x01} (sesión por defecto) y \texttt{0x03} (sesión extendida). La respuesta positiva (\texttt{0x50}) incluye los tiempos de servidor \texttt{P2} (25\,ms) y \texttt{P2*} (500\,ms). Cualquier otra subfunción se rechaza con NRC \texttt{0x12}.

    \item \textbf{Servicio \texttt{0x22} — ReadDataByIdentifier}: \texttt{handleMode22()} responde a los DID estándar \texttt{0xF190} (VIN), \texttt{0xF18C} (número de serie de la ECU) y \texttt{0xF189} (versión de software), accesibles en cualquier sesión, y a los DID propietarios de telemetría \texttt{0x2001}--\texttt{0x2008} (carga, refrigerante, RPM, velocidad, acelerador, combustible, aceite y batería), accesibles \textbf{solo en sesión extendida}. Una petición de un DID propietario fuera de la sesión extendida se rechaza con NRC \texttt{0x22} (\textit{conditionsNotCorrect}), y un DID desconocido con NRC \texttt{0x31} (\textit{requestOutOfRange}). Las respuestas que superan 7 bytes (como el VIN) se transmiten mediante la secuencia ISO-TP multi-frame genérica \texttt{sendMultiFrame()}.
\end{itemize}

Ambos servicios se implementaron manualmente, sin recurrir a ninguna librería UDS. El Listado~\ref{lst:handle_mode10} muestra el control de sesión (\texttt{0x10}): la respuesta positiva \texttt{0x50} devuelve los tiempos \texttt{P2}/\texttt{P2*} y cualquier subfunción distinta de \texttt{0x01}/\texttt{0x03} se rechaza con NRC.

\begin{lstlisting}[language=C++,caption={Servicio UDS 0x10 — DiagnosticSessionControl (\texttt{handleMode10()}).},label={lst:handle_mode10}]
bool handleMode10(uint8_t subFunc) {
  if (subFunc != 0x01 && subFunc != 0x03) {            // solo default / extended
    sendNegativeResponse(0x10, NRC_SUBFUNCTION_NOT_SUPP);
    return true;
  }
  udsSession = subFunc;                                // cambia el estado de sesión
  responseBuffer[0] = 0x50;                            // respuesta positiva (0x10 + 0x40)
  responseBuffer[1] = subFunc;
  responseBuffer[2] = 0x00; responseBuffer[3] = 0x19;  // P2   = 25 ms
  responseBuffer[4] = 0x01; responseBuffer[5] = 0xF4;  // P2*  = 500 ms
  responseLength = 6;
  sendResponse();
  return true;
}
\end{lstlisting}

El Listado~\ref{lst:handle_mode22} recoge la lectura de datos por identificador (\texttt{0x22}). Se distingue entre DID estándar accesibles en cualquier sesión y DID propietarios que exigen sesión extendida: si no se cumple esa condición se responde con NRC \texttt{0x22} (\textit{conditionsNotCorrect}) y, si el DID no existe, con NRC \texttt{0x31} (\textit{requestOutOfRange}).

\begin{lstlisting}[language=C++,caption={Servicio UDS 0x22 — ReadDataByIdentifier (\texttt{handleMode22()}, fragmento).},label={lst:handle_mode22}]
bool handleMode22(uint16_t did) {
  uint8_t payload[24];
  uint8_t payloadLen = 0;
  payload[0] = 0x62;                       // respuesta positiva (0x22 + 0x40)
  payload[1] = (did >> 8) & 0xFF;
  payload[2] = did & 0xFF;

  switch (did) {
    case 0xF190:                           // VIN: DID estándar, cualquier sesión
      for (uint8_t i = 0; i < VIN_LENGTH; i++) payload[3 + i] = vehicle.vin[i];
      payloadLen = 3 + VIN_LENGTH;
      break;
    case 0x2003:                           // RPM: DID propietario, solo extendida
      if (udsSession != 3) {
        sendNegativeResponse(0x22, NRC_CONDITIONS_NOT_CORRECT);   // NRC 0x22
        return true;
      }
      encodeRPM(&payload[3], vehicle.rpm);
      payloadLen = 5;
      break;
    // ... resto de DID 0x2001-0x2008 (misma comprobación de sesión) ...
    default:
      sendNegativeResponse(0x22, NRC_REQUEST_OUT_OF_RANGE);       // NRC 0x31
      return true;
  }

  if (payloadLen <= ISOTP_SF_MAX_PAYLOAD) {            // cabe en Single Frame
    memcpy(responseBuffer, payload, payloadLen);
    responseLength = payloadLen;
    sendResponse();
  } else {
    sendMultiFrame(payload, payloadLen);              // multi-frame (p. ej. VIN)
  }
  return true;
}
\end{lstlisting}


\subsection{Mecanismos opcionales de variabilidad del bus}
\label{sec:ruido_can}

Para poder validar la herramienta de diagnóstico tanto en condiciones ideales como realistas, el simulador puede operar de forma \textbf{determinista} (configuración por defecto, bus silencioso salvo las respuestas a peticiones) o activar dos mecanismos opcionales de variabilidad mediante macros de compilación en \texttt{ecu\_protocol.h}.

\medskip
\noindent\textbf{Ruido en los sensores (\texttt{ADD\_NOISE})}

Cuando \texttt{ADD\_NOISE} se compila a \texttt{1} (por defecto \texttt{0}), el simulador añade \textit{jitter} pseudoaleatorio a los valores de los sensores en cada ciclo de actualización. La amplitud máxima de cada sensor se parametriza de forma independiente mediante constantes (\texttt{NOISE\_RPM\_MAX} = 25\,rpm, \texttt{NOISE\_VOLTAGE\_MAX} = 30\,mV, \texttt{NOISE\_MAF\_MAX} = 8, etc.). Tras añadir el ruido, cada variable se acota a su rango físico válido, de modo que la telemetría conserva un comportamiento creíble pero no perfectamente estable, reproduciendo la lectura real de sensores analógicos.

\medskip
\noindent\textbf{Tramas de difusión (\texttt{BROADCAST\_ENABLE})}

Cuando \texttt{BROADCAST\_ENABLE} se compila a \texttt{1} (por defecto \texttt{0}), el simulador emite cada \texttt{BROADCAST\_INTERVAL\_MS} (100\,ms) tres tramas CAN no solicitadas que imitan el tráfico periódico de una ECU real: \texttt{0x280} (RPM y carga del motor), \texttt{0x320} (velocidad) y \texttt{0x3D0} (temperatura de refrigerante, acelerador y tensión de batería). Este tráfico de fondo permite verificar que el filtro de la herramienta de diagnóstico descarta correctamente los identificadores ajenos a la dirección de respuesta de la ECU (\texttt{0x7E8}) y no los interpreta como respuestas válidas.


\subsection{Inyección de DTCs por interrupción hardware}

Para simular la aparición espontánea de averías durante una sesión de diagnóstico, el simulador implementa un mecanismo de inyección de DTCs activado por la señal en el pin digital D4 (\texttt{DTC\_FAULT\_PIN}).

El patrón de implementación sigue la práctica estándar para ISR en Arduino:

\begin{enumerate}
    \item La ISR \texttt{onDTCPress()} es minimalista: aplica una histéresis temporal de 300 ms (\texttt{DTC\_HYSTERESIS\_MS}) mediante \texttt{millis()} para el prevenir el rebote e incrementa el contador \texttt{volatile uint16\_t dtcFlag}.
    \item La inyección propiamente dicha se realiza en \texttt{handleDTCButton()}, llamada desde el bucle principal, donde es seguro ejecutar operaciones con efectos de bloqueo (\texttt{Serial}).
    \item Cada pulsación inserta secuencialmente el siguiente código del conjunto \texttt{DTC\_FAULT\_SET} (\texttt{P0300} (fallo de encendido), \texttt{P0171} (mezcla pobre) y \texttt{P0420} (eficiencia del catalizador)) en \texttt{dtcList[]} y activa el testigo \texttt{checkEngine}.
    \item Cuando ya hay tres averías activas, una pulsación adicional borra todas, permitiendo reiniciar el ciclo de prueba.
\end{enumerate}

El Listado~\ref{lst:isr_dtc} muestra el patrón ISR mínima con histéresis: la rutina de interrupción solo aplica el antirrebote por \texttt{millis()} e incrementa un contador \texttt{volatile}, dejando todo el trabajo pesado para el bucle principal.

\begin{lstlisting}[language=C++,caption={Inyección de DTCs por interrupción: ISR mínima con histéresis (\texttt{onDTCPress()}) y procesamiento diferido (\texttt{handleDTCButton()}).},label={lst:isr_dtc}]
volatile uint16_t dtcFlag    = 0;
volatile uint32_t lastDtcISR = 0;

// ISR minimalista: antirrebote por millis() + incremento de un contador volatil
void onDTCPress() {
  uint32_t now = millis();
  if ((uint32_t)(now - lastDtcISR) < DTC_HYSTERESIS_MS) return;   // 300 ms
  lastDtcISR = now;
  dtcFlag++;
}

// Trabajo pesado fuera de la ISR, invocado desde loop() (Serial es seguro aqui)
void handleDTCButton() {
  if (!dtcFlag) return;
  dtcFlag--;
  if (numStoredDTCs >= DTC_FAULT_SET_SIZE) {       // cuarta pulsacion: borra todo
    for (uint8_t i = 0; i < 8; i++) { dtcList[i].active = false; }
    numStoredDTCs = 0; vehicle.numDTCs = 0; vehicle.checkEngine = false;
  } else {                                         // inyecta el siguiente DTC del set
    dtcList[numStoredDTCs].code   = DTC_FAULT_SET[numStoredDTCs];
    dtcList[numStoredDTCs].active = true;
    numStoredDTCs++;
    vehicle.numDTCs = numStoredDTCs; vehicle.checkEngine = true;
  }
}
\end{lstlisting}

Esta funcionalidad permite al operador del banco de pruebas introducir averías de forma controlada durante una sesión de diagnóstico activa y verificar que la herramienta Python las detecta correctamente en la siguiente consulta del modo 0x03.

\section{Herramienta de diagnóstico — Raspberry Pi}
\label{sec:raspberry}

La herramienta de diagnóstico es el componente central del sistema: recibe peticiones desde la aplicación móvil vía BLE, las traduce a secuencias ISO-TP sobre el bus CAN, interpreta las respuestas de la ECU, almacena los datos en SQLite y devuelve los resultados al cliente móvil. Se implementa en Python 3 sobre Raspberry Pi con controlador CAN MCP2515. El código fuente está disponible en el repositorio del proyecto.


\subsection{Arquitectura software}
\label{sec:python_arch}

La herramienta sigue un diseño basado en interfaces y capas desacopladas mediante inyección de dependencias. Esta decisión, supuso una inversión inicial de tiempo mayor a la contemplada en el diseño rápido pero resultó acertada pues en las fases posteriores facilitó las pruebas sin hardware y la extensión progresiva del sistema.

Las interfaces abstractas principales son:

\begin{itemize}
    \item \textbf{\texttt{ITransport}}: define las operaciones de conexión y de envío/recepción de mensajes ISO-TP (\texttt{connect}, \texttt{disconnect}, \texttt{send}, \texttt{receive}). Las implementaciones concretas son \texttt{IsoTpTransport} (bus CAN real mediante \texttt{python-can} + \texttt{can-isotp} sobre SocketCAN) y \texttt{MockTransport} (respuestas predefinidas para pruebas sin hardware).

    \item \textbf{\texttt{IProtocolBuilder}}: define la construcción de las tramas de petición OBD-II mediante métodos específicos por operación (\texttt{build\_read\_rpm\_request}, \texttt{build\_read\_dtcs\_request}, \texttt{build\_read\_vin\_request}, etc.). La implementación \texttt{Obd2ProtocolBuilder} las genera según el estándar SAE J1979.

    \item \textbf{\texttt{IDataDecoder}}: valida la trama de respuesta (\texttt{validate\_response}, que detecta respuestas negativas NRC) y la decodifica por parámetro (\texttt{decode\_rpm}, \texttt{decode\_coolant\_temp}, \texttt{decode\_dtcs}, \texttt{decode\_vin}, etc.). La implementación \texttt{Obd2DataDecoder} aplica las fórmulas del estándar para cada PID y devuelve valores con unidades.

    \item \textbf{\texttt{IDiagnosticSession}}: define la interfaz de sesión de diagnóstico (\texttt{get\_engine\_rpm}, \texttt{get\_dtcs}, \texttt{clear\_dtcs}, \texttt{get\_vin}, \texttt{get\_snapshot}, etc.). La implementación \texttt{DiagnosticSession} orquesta las demás interfaces ejecutando la secuencia enviar\,→\,recibir\,→\,validar\,→\,decodificar.
\end{itemize}

Sobre esta estructura base se aplica el patrón decorador en dos niveles. \textbf{\texttt{LoggingTransport}} envuelve cualquier \texttt{ITransport} e intercepta todas las operaciones de envío y recepción para volcarlas al registro textual de comunicaciones (\texttt{ble\_comms.log}) y mostrar la última trama enviada/recibida. A su vez, \textbf{\texttt{LoggedDiagnosticSession}} envuelve la \texttt{DiagnosticSession} y, tras cada operación, almacena en la base de datos SQLite el comando ejecutado junto con sus tramas de petición y respuesta (y los DTCs leídos). La composición resultante es:

\[
    \texttt{LoggedDiagnosticSession}\big(
    \texttt{DiagnosticSession}(\texttt{LoggingTransport}(\texttt{IsoTpTransport}),
\]
\[
    \texttt{Obd2ProtocolBuilder},\ \texttt{Obd2DataDecoder}),\ \texttt{SqliteDataLogger}\big)
\]

El núcleo funcional de la herramienta es el ciclo de consulta de un PID, que orquesta las cuatro interfaces en la secuencia construir\,→\,enviar\,→\,recibir\,→\,validar\,→\,decodificar. El Listado~\ref{lst:consulta_pid} muestra este flujo en la clase \texttt{DiagnosticSession}, común a todas las operaciones del modo 0x01; cada método público sigue exactamente el mismo patrón de cuatro pasos.

\begin{lstlisting}[language=Python,caption={Ciclo de consulta de un PID en \texttt{DiagnosticSession} (construir → enviar → recibir → validar → decodificar).},label={lst:consulta_pid}]
def get_engine_rpm(self) -> float:
    self._transport.send(self._builder.build_read_rpm_request())  # construir + enviar
    raw = self._transport.receive()                               # recibir (ISO-TP reensambla)
    self._decoder.validate_response(raw, expected_mode=0x01)      # validar (detecta NRC 0x7F)
    return self._decoder.decode_rpm(raw)                          # decodificar (formula J1979)

def get_dtcs(self) -> list[Dtc]:
    self._transport.send(self._builder.build_read_dtcs_request())
    raw = self._transport.receive()
    self._decoder.validate_response(raw, expected_mode=0x03)
    return self._decoder.decode_dtcs(raw)
\end{lstlisting}

La decodificación de los DTC (modo 0x03) es un ejemplo de implementación propia del formato SAE J2012. El Listado~\ref{lst:decode_dtcs} muestra cómo \texttt{decode\_dtcs()} recorre los pares de bytes y cómo \texttt{Dtc.from\_raw()} extrae la categoría (P/C/B/U) de los dos bits más significativos y los cuatro dígitos hexadecimales restantes.

\begin{lstlisting}[language=Python,caption={Decodificación de DTCs en formato SAE J2012 (\texttt{decode\_dtcs()} y \texttt{Dtc.from\_raw()}).},label={lst:decode_dtcs}]
# Bits 15-14 de la palabra de 2 bytes -> prefijo de categoria
_DTC_PREFIX = {0b00: "P", 0b01: "C", 0b10: "B", 0b11: "U"}

def decode_dtcs(self, raw: bytes) -> list[Dtc]:
    dtc_count = raw[1]                       # num. de DTC en la respuesta
    dtcs = []
    for i in range(dtc_count):
        offset = 2 + i * 2
        pair = raw[offset: offset + 2]       # cada DTC ocupa 2 bytes
        if pair == b"\x00\x00":
            continue
        dtcs.append(Dtc.from_raw(pair))
    return dtcs

@classmethod
def from_raw(cls, raw: bytes, lang: str = "es") -> "Dtc":
    prefix = _DTC_PREFIX[(raw[0] >> 6) & 0x03]   # categoria (P/C/B/U)
    digit1 = (raw[0] >> 4) & 0x03
    digit2 = raw[0] & 0x0F
    digit3 = (raw[1] >> 4) & 0x0F
    digit4 = raw[1] & 0x0F
    code = f"{prefix}{digit1}{digit2:X}{digit3:X}{digit4:X}"   # p. ej. P0300
    ...
\end{lstlisting}


\subsection{Capa de transporte CAN e ISO-TP}

La comunicación con el bus CAN se realiza a través de la interfaz \texttt{SocketCAN} de Linux, configurada mediante \texttt{ip link set can0 up type can bitrate 500000}. La librería \texttt{python-can} proporciona una abstracción de alto nivel sobre esta interfaz, mientras que \texttt{can-isotp} gestiona de forma automática la segmentación y reensamblado ISO-TP, el control de flujo y la temporización entre tramas.

La configuración del socket ISO-TP (\texttt{isotp.Address} en modo \textit{Normal 11 bits}) define la dirección del escáner (\texttt{txid = 0x7E0}) y la dirección de respuesta de la ECU (\texttt{rxid = 0x7E8}), con \texttt{tx\_padding = 0xAA} para el relleno a 8 bytes, \texttt{blocksize = 0} (se aceptan todas las \textit{Consecutive Frames}) y \texttt{stmin = 25\,ms}. El timeout de recepción se fija en 5\,segundos, valor holgado para absorber la latencia de la transmisión multi-frame sin generar falsos errores.


\subsection{Módulo de logging en SQLite}

Toda la información de diagnóstico se almacena en una base de datos SQLite cuyo esquema se describe a continuación. La base de datos opera en modo WAL (\textit{Write-Ahead Logging}), que permite escrituras concurrentes desde el hilo de monitorización y el hilo de atención a comandos BLE sin bloqueos mutuos.

Las cuatro tablas del esquema son:

\begin{itemize}
    \item \textbf{\texttt{sessions}}: registra cada sesión de diagnóstico con una etiqueta (\texttt{label}) y los \textit{timestamps} de inicio y fin (\texttt{started\_at}, \texttt{ended\_at}). El número de muestras de cada sesión no se almacena, sino que se calcula mediante una consulta agregada sobre la tabla \texttt{samples}.

    \item \textbf{\texttt{samples}}: almacena cada muestra de telemetría con el PID, el nombre, el valor decodificado, la unidad, un \textit{timestamp} monotónico (\texttt{monotonic\_ts}) y un \textit{timestamp} de pared en formato ISO-8601 (\texttt{wall\_ts}).

    \item \textbf{\texttt{commands}}: registra cada comando ejecutado con su nombre (\texttt{command}), las tramas de petición y respuesta en hexadecimal (\texttt{request\_hex}, \texttt{response\_hex}) y el \textit{timestamp}. Estos campos permiten reconstruir la secuencia completa de comunicación ISO-TP para análisis forense.

    \item \textbf{\texttt{dtcs}}: almacena los códigos de avería leídos en cada sesión (código en formato J2012, bytes en bruto, descripción y \textit{timestamp}).
\end{itemize}

El \texttt{SqliteDataLogger} acumula las muestras en un buffer en memoria y las vuelca a SQLite en lotes de 50 (\texttt{\_BUFFER\_SIZE}), reduciendo la frecuencia de escritura a disco y mejorando el rendimiento general del sistema durante sesiones de monitorización prolongadas.

\subsection{Monitor de datos en tiempo real}

El monitor de datos (\texttt{LiveDataMonitor}) se ejecuta en un hilo \textit{daemon} en segundo plano con un periodo de muestreo de 500\,ms por defecto. En cada ciclo consulta un conjunto configurable de PIDs del modo 0x01, decodifica los valores recibidos e invoca un \textit{callback} \texttt{on\_sample} por cada muestra. El \texttt{BtCommandHandler} agrupa las muestras de cada ciclo y las reenvía al cliente móvil como una única notificación BLE, además de almacenarlas en SQLite. El acceso al bus CAN se serializa con el hilo de atención a comandos mediante un \textit{lock}(candado de mutua exclusión) compartido.
El conjunto de PIDs monitorizados por defecto incluye temperatura del refrigerante, carga calculada del motor, RPM, velocidad y posición del acelerador. La lista es configurable en tiempo de ejecución a través del comando JSON correspondiente.

La sincronización entre el hilo de monitorización y el hilo de atención a comandos BLE es uno de los puntos críticos del diseño: ambos comparten el mismo bus CAN, por lo que una petición puntual (por ejemplo, leer los DTC) no puede solaparse con el ciclo de muestreo continuo. La solución consiste en inyectar un único \texttt{threading.Lock} compartido a ambos componentes. El Listado~\ref{lst:lock} muestra el ciclo de muestreo del monitor, que adquiere ese mismo \textit{lock} para enviar la petición, recibir la respuesta y descartar las tramas tardías de peticiones anteriores antes de liberarlo.

\begin{lstlisting}[language=Python,caption={Acceso serializado al bus CAN desde el hilo de monitorización mediante el lock compartido (\texttt{LiveDataMonitor.\_poll\_single}).},label={lst:lock}]
def _poll_single(self, pid_def):
    if pid_def.pid in self._nrc_pids:          # PID ya marcado como no soportado
        return
    with self._lock:                           # mismo lock que el hilo de comandos BLE
        self._transport.send(pid_def.request)
        raw = self._transport.receive()
        # descarta tramas tardias de peticiones previas sin soltar el lock
        for _ in range(self._MAX_DRAIN):
            if len(raw) < 2 or raw[0] == 0x7F or raw[1] == pid_def.pid:
                break
            raw = self._transport.receive()
    # ... validacion y decodificacion fuera de la seccion critica ...
\end{lstlisting}

El mismo \texttt{self.\_lock} es el que adquiere \texttt{BtCommandHandler} en cada comando que accede al bus (\texttt{with self.\_lock: self.\_session.get\_dtcs()}), de modo que las peticiones puntuales y el muestreo continuo nunca compiten por el transporte.

\subsection{Servidor BLE GATT}

El servidor BLE se implementa mediante la librería \texttt{bless}, que proporciona una interfaz asíncrona (\texttt{asyncio}) para la gestión del servicio GATT. El servidor expone el servicio NUS con sus dos características (TX y RX) y gestiona la conexión del cliente BLE. El Listado~\ref{lst:ble_init} muestra la publicación del servicio NUS: la característica RX se declara con permisos de escritura (recepción de comandos) y la TX con la propiedad \textit{notify} (envío de respuestas y telemetría).

El manejador de comandos \texttt{BtCommandHandler} recibe los objetos JSON escritos por el cliente en la característica RX, los despacha al método correspondiente (sobre la \texttt{LoggedDiagnosticSession}, el logger o el transporte según el comando) y envía la respuesta JSON como notificación en la característica TX.

Se implementa un mecanismo de autenticación por token: el primer comando de cada sesión BLE debe incluir el campo \texttt{token} con el valor configurado en el servidor. Las conexiones no autenticadas reciben una respuesta de error y se cierran tras un timeout de inactividad.

Los comandos JSON implementados son:

\begin{itemize}
    \item \verb|{"cmd": "auth", "token": "..."}|: autenticación de sesión.
    \item \verb|{"cmd": "ping"}|: comprobación de enlace (responde \texttt{pong}).
    \item \verb|{"cmd": "snapshot"}|: lectura puntual de todos los PIDs soportados del modo 0x01.
    \item \verb|{"cmd": "dtcs"}| / \verb|{"cmd": "clear_dtcs"}|: lectura y borrado de DTCs.
    \item \verb|{"cmd": "vin"}|: obtención del VIN del vehículo.
    \item \verb|{"cmd": "monitor_start", "pids": [12,13,5]}| / \verb|{"cmd": "monitor_stop"}|: inicio y detención de la monitorización continua.
    \item \verb|{"cmd": "probe_pids"}|: descubrimiento automático de los PIDs soportados por la ECU.
    \item \verb|{"cmd": "sessions"}|, \verb|{"cmd": "session_samples", "session_id": N}|, \verb|{"cmd": "session_commands", "session_id": N}|, \verb|{"cmd": "session_dtcs", "session_id": N}|: consulta del historial SQLite.
    \item \verb|{"cmd": "uds_session", "session_type": 3}| / \verb|{"cmd": "uds_read_did", "did": "0xF190"}|: control de sesión UDS y lectura de DIDs.
    \item \verb|{"cmd": "disconnect"}|: cierre ordenado de la sesión.
\end{itemize}

Los comandos \texttt{uds\_session} y \texttt{uds\_read\_did} se atienden a través de un cliente UDS propio (\texttt{UdsSession}). Una decisión de diseño relevante es que la comprobación de sesión se realiza \textbf{localmente} antes de acceder al bus: si se solicita un DID propietario sin estar en sesión extendida, el cliente lanza la excepción correspondiente sin llegar a emitir la petición CAN, como muestra el Listado~\ref{lst:uds_client}.

\begin{lstlisting}[language=Python,caption={Cliente UDS: control de sesión y lectura de DID con validación local de sesión (\texttt{UdsSession}).},label={lst:uds_client}]
def set_session(self, session_type: int) -> dict:
    req = self._builder.build_session_control(session_type)
    self._transport.send(req)
    raw = self._transport.receive()
    response = self._decoder.validate_response(raw, _SID_SESSION_CTRL)
    info = self._decoder.decode_session_info(response)
    self._current_session = session_type          # recuerda la sesion activa
    return info

def read_did(self, did: int) -> object:
    definition = _dids.DIDS.get(did)
    if definition is None:
        raise InvalidResponseError(f"DID 0x{did:04X} not in registry")
    # validacion LOCAL: no se accede al bus si la sesion no es la correcta
    if definition.extended_only and self._current_session != _dids.UDS_SESSION_EXTENDED:
        raise NrcException(mode=_SID_READ_DATA_BY_ID, nrc_code=0x7E)

    self._transport.send(req)
    raw = self._transport.receive()

    return self._decoder.decode_did(response, did)

El comando \verb|probe_pids| merece una mención aparte por su lógica en dos etapas: primero declara los PIDs soportados leyendo las máscaras de bits estándar (\texttt{0x00}, \texttt{0x20}, \dots), y después confirma cada uno con una consulta real para descartar falsos positivos. El Listado~\ref{lst:probe_pids} recoge la implementación real de este procedimiento.

\begin{lstlisting}[language=Python,caption={Descubrimiento de PIDs soportados en dos etapas (\texttt{\_probe\_pids}).},label={lst:probe_pids}]
def _probe_pids(self, _cmd: dict) -> dict:
    # Etapa 1: PIDs declarados via mascaras de disponibilidad
    declared: set[int] = set()
    for support_pid in (0x00, 0x20, 0x40, 0x60, 0x80, 0xA0, 0xC0, 0xE0):
        try:
            with self._lock:
                self._transport.send(bytes([0x01, support_pid]))
                raw = self._transport.receive()
            if len(raw) >= 6 and raw[0] == 0x41 and raw[1] == support_pid:
                mask = (raw[2] << 24) | (raw[3] << 16) | (raw[4] << 8) | raw[5]
                for i in range(32):
                    if mask & (0x80000000 >> i):
                        declared.add(support_pid + i + 1)
                if not (mask & 0x01):            # no se anuncia el siguiente rango
                    break
        except Exception:
            break

    # Etapa 2: confirmar cada PID con una consulta real (descarta falsos positivos)
    confirmed: set[int] = set()
    for pid in sorted(declared):
        try:
            with self._lock:
                self._transport.send(bytes([0x01, pid]))
                raw = self._transport.receive()
            if len(raw) >= 2 and raw[0] == 0x41 and raw[1] == pid:
                confirmed.add(pid)
        except Exception:
            pass

    # solo se exponen los PIDs que tienen decodificador
    self._supported_pids = confirmed & set(PIDS.keys())
    return {"status": "ok", "data": sorted(self._supported_pids)}
\end{lstlisting}

\section{Aplicación móvil — React Native}
\label{sec:app_movil}

La aplicación móvil es la interfaz de usuario del sistema de diagnóstico. Se implementa con React Native y Expo, lo que permite compartir una única base de código entre Android e iOS. El código fuente está disponible en el repositorio del proyecto.

\subsection{Arquitectura de la aplicación}

La aplicación sigue una arquitectura basada en tres capas:

\begin{enumerate}
    \item \textbf{Capa de adaptadores}: abstrae el acceso al hardware BLE mediante la interfaz \texttt{IVehicleAdapter}. La implementación concreta \texttt{BleAdapter} utiliza la librería \texttt{react-native-ble-plx} para las operaciones de escaneo, conexión, escritura y suscripción a notificaciones. La implementación \texttt{MockAdapter} simula un dispositivo BLE para el desarrollo sin hardware, con respuestas predefinidas para todos los comandos. La fábrica \texttt{adapterFactory} selecciona una u otra según el modo de la aplicación.

    \item \textbf{Capa de estado}: gestiona el estado global de la aplicación mediante almacenes Zustand. Los almacenes principales son \texttt{useConnectionStore} (estado de la conexión BLE y dispositivo activo), \texttt{useDashboardStore} y \texttt{useVehicleStore} (últimos valores de los PIDs monitorizados), \texttt{useDtcStore} (lista de DTCs), \texttt{useSessionStore} (historial de sesiones y muestras), \texttt{useUdsStore} (sesión y DIDs UDS), \texttt{usePidSupportStore} (PIDs soportados tras el sondeo) y \texttt{useSettingsStore} (modo simulación y preferencias).

    \item \textbf{Capa de presentación}: las cinco pantallas de la aplicación consumen el estado de los almacenes Zustand y despachan comandos a través del adaptador BLE.
\end{enumerate}

\subsection{Pantallas y funcionalidades}

La navegación principal se organiza en cinco pestañas (\textit{bottom tab navigator}): \textit{Panel}, \textit{Averías}, \textit{Sesiones}, \textit{Consola} y \textit{Ajustes}. El flujo de conexión (escaneo, selección de dispositivo y autenticación) se presenta mediante diálogos modales lanzados desde la pantalla de Ajustes o desde el estado vacío de las pantallas operativas. A continuación se detallan los casos de uso de cada pantalla y los controles que los soportan.

\noindent\textbf{Estado sin conexión}

Mientras no hay conexión activa ni modo de simulación, las pantallas operativas (\textit{Panel}, \textit{Averías} y \textit{Consola}) muestran un estado vacío contextual que informa de que el usuario está sin conexión y describe la función de la pantalla. Desde este estado se ofrecen dos botones: \textbf{CONECTAR}, que abre el modal de búsqueda de dispositivos, y \textbf{SIMULAR}, que activa el modo de simulación.

\noindent\textbf{Flujo de conexión}

El modal de búsqueda realiza un escaneo BLE de hasta 20 segundos (botones \textit{Buscar}/\textit{Detener}/\textit{Actualizar} y \textit{Cancelar}) y enumera todos los dispositivos cercanos, situando en primer lugar el adaptador objetivo (la Raspberry Pi se anuncia como \texttt{diag\_tool}), que se marca con la etiqueta \textit{Recomendado}. Para cada dispositivo se indica una estimación cualitativa de la intensidad de señal RSSI traducida a etiquetas legibles (\textit{Señal excelente}, \textit{buena}, \textit{débil} o \textit{muy débil}) y un botón \textit{Conectar}. Al seleccionar un dispositivo, la aplicación recorre las fases de conexión física, negociación de MTU, descubrimiento de servicios, suscripción a notificaciones y autenticación por token, cerrando el modal automáticamente al quedar conectada y solicitando entonces el VIN del vehículo y los PIDs soportados. El adaptador BLE implementa un \textit{watchdog} de inactividad que comprueba la actividad cada 5\,segundos y desconecta la sesión si esta cesa.

\subsection{Comunicación BLE}

La librería \texttt{react-native-ble-plx} gestiona las operaciones BLE de bajo nivel. La característica RX del servicio NUS se escribe mediante operaciones \texttt{writeCharacteristicWithResponse}, que garantizan la entrega del comando antes de esperar la respuesta. La característica TX se monitoriza mediante \texttt{monitorCharacteristicForService}, que invoca un callback cada vez que la Raspberry Pi envía una notificación.

El protocolo se basa en mensajes JSON delimitados por saltos de línea (NDJSON): tanto la Raspberry Pi como la aplicación fragmentan los mensajes que superan el tamaño de escritura BLE (240 bytes) y los reensamblan en el receptor acumulando los fragmentos hasta el delimitador \texttt{\textbackslash n}. La aplicación negocia un MTU de 128 bytes en la conexión. En la práctica, la fragmentación solo se produce en las respuestas de historial de sesiones con un número elevado de muestras.

El receptor debe resolver dos problemas simultáneamente: reensamblar las notificaciones BLE (que pueden llegar fragmentadas o agrupar varios mensajes) y emparejar cada respuesta con su petición sin bloquear el flujo de muestras de la monitorización continua. El Listado~\ref{lst:ndjson} muestra la lógica real del adaptador: un búfer de recepción acumula los fragmentos y los separa por el delimitador \texttt{\textbackslash n}, mientras que un despachador encamina cada mensaje según su tipo. Los mensajes asíncronos (muestras de telemetría y \textit{heartbeats}) se entregan a los \textit{callbacks} del monitor o se responden de inmediato; el resto se empareja con la petición pendiente más antigua de una cola FIFO, que resuelve o rechaza la promesa correspondiente. La implementación completa está disponible en el repositorio del proyecto.
% ============================================================================

\begin{lstlisting}[language=JavaScript,caption={Reensamblado NDJSON y emparejamiento petición-respuesta en el adaptador BLE (React Native).},label={lst:ndjson}]
// Buffer de recepcion: acumula fragmentos y los separa por '\n'
private feedBuffer(chunk: string): void {
  this.rxBuf += chunk;
  const lines = this.rxBuf.split('\n');
  this.rxBuf = lines.pop() ?? '';          // ultimo trozo = linea incompleta
  for (const line of lines) {
    const t = line.trim();
    if (!t) continue;
    try {
      this.dispatch(JSON.parse(t));         // JSON malformado -> se ignora
    } catch { /* skip */ }
  }
}

private dispatch(msg: Record<string, unknown>): void {
  // Mensajes asincronos: se entregan a los callbacks del monitor
  if (msg.type === 'samples') {
    (msg.samples as MonitorSample[]).forEach(
      (s) => this.sampleCbs.forEach((cb) => cb(s)));
    return;
  }
  if (msg.type === 'heartbeat') { this.sendHeartbeatAck(); return; }

  // Respuesta a comando: se empareja con la peticion mas antigua (cola FIFO)
  const pending = this.queue.shift();
  if (!pending) return;
  clearTimeout(pending.timer);
  if (msg.status === 'ok') pending.resolve(msg.data);
  else pending.reject(new Error((msg.message as string) ?? 'Server error'));
}
\end{lstlisting}

\section{Integración y pruebas}
\label{sec:integracion}

La validación del sistema completo se realizó sobre el banco de ensayo formado por el Arduino MKR con shield CAN, la Raspberry Pi con controlador MCP2515 y una tableta Android con la aplicación instalada. Los tres nodos se conectaron mediante un bus CAN de dos hilos con terminación de 120\,$\Omega$ en cada extremo.

\subsection{Escenarios de prueba}

Se definieron los siguientes escenarios de prueba que cubren los flujos principales del sistema y los casos de error más relevantes:

\textbf{Escenario 1 — Ciclo OBD-II básico.} Verificación del ciclo completo de consulta de un PID del modo 0x01 (RPM, \texttt{0x0C}): envío de la petición desde la aplicación móvil, transmisión CAN de la petición ISO-TP, respuesta del simulador con el valor codificado, decodificación en la Raspberry Pi y presentación en el Panel.

\textbf{Escenario 2 — Respuesta VIN multi-frame.} Verificación de la secuencia ISO-TP completa para la consulta del VIN (\texttt{0x09 0x02}): First Frame, Flow Control y dos Consecutive Frames. Se comprobó que el VIN reconstruido coincide exactamente con la cadena configurada en el simulador y que las tramas CAN correspondientes se registran en la base de datos.

\textbf{Escenario 3 — Gestión de DTCs.} Verificación del ciclo completo de gestión de averías: lectura de DTCs inicial (lista vacía), inyección de DTCs mediante la interrupción hardware del pin D4, lectura tras la inyección (DTCs presentes), borrado de DTCs mediante la aplicación y lectura final (lista vacía de nuevo).

\textbf{Escenario 4 — Monitorización continua prolongada.} Sesión de monitorización de 10 minutos con cinco PIDs activos (refrigerante, carga, RPM, velocidad y acelerador) a 500\,ms de periodo. Se verificó la estabilidad de la conexión BLE, la ausencia de pérdidas de muestras y el correcto comportamiento del sistema bajo carga sostenida.

\textbf{Escenario 5 — Sesión y DIDs UDS.} Verificación del cambio de sesión UDS (\texttt{0x10}) a sesión extendida y la lectura de los DID estándar y propietarios (\texttt{0x22}). Se comprobó que los DID propietarios solo responden en sesión extendida y que, en sesión por defecto, el simulador devuelve la respuesta negativa NRC \texttt{0x22}, que la herramienta registra sin bloquear la sesión.

\textbf{Escenario 6 — Robustez ante variabilidad del bus.} Con los mecanismos opcionales del simulador activos (\texttt{ADD\_NOISE} y \texttt{BROADCAST\_ENABLE}), se comprobó que las tramas de difusión con identificadores ajenos (\texttt{0x280}, \texttt{0x320}, \texttt{0x3D0}) son descartadas por la pila ISO-TP sin interferir en la comunicación OBD-II, y que el ruido en los sensores no impide la correcta decodificación de los valores.

\textbf{Escenario 7 — Reconexión BLE.} Verificación del comportamiento ante la pérdida de la conexión Bluetooth: el adaptador detecta la desconexión, libera los recursos y notifica al usuario, que puede reiniciar la conexión sin pérdida de los datos históricos persistidos en SQLite.

\subsection{Resultados de las pruebas de integración}

Todos los escenarios de prueba definidos se completaron con resultado satisfactorio en el banco de ensayo.

La principal dificultad encontrada durante las pruebas de integración fue la reconexión BLE ante caídas inesperadas del enlace por parte de la Raspberry Pi. Cuando la Pi cerraba la conexión de forma abrupta, el sistema operativo Android conservaba un descriptor GATT obsoleto, de modo que el siguiente intento de conexión fallaba con un error de \textit{dispositivo ya conectado}. Además, las peticiones que quedaban a la espera de respuesta dejaban la interfaz bloqueada indefinidamente. La corrección consistió en tres medidas: registrar un \textit{callback} \texttt{onDisconnected} que libera todos los recursos (intervalos de \textit{ping}, suscripción a notificaciones, búfer de recepción) y rechaza las peticiones pendientes de la cola para desbloquear la interfaz, cancelar explícitamente cualquier conexión GATT residual con \texttt{cancelDeviceConnection()} antes de cada nuevo intento y un \textit{watchdog} de inactividad que fuerza la desconexión limpia si no se recibe actividad durante 20 s, evitando que la aplicación quede atrapada en un estado de conexión fantasma. Tras estos cambios, el usuario puede reiniciar la conexión sin reiniciar la aplicación.

Las pruebas con los mecanismos opcionales de variabilidad activos confirmaron que el filtro de la pila ISO-TP descarta correctamente las tramas de difusión ajenas a la dirección de respuesta de la ECU, y que el timeout de recepción de 5 segundos absorbe con holgura la latencia de la transmisión multi-frame, manteniendo una tasa de errores no recuperables despreciable en todas las sesiones de prueba.
